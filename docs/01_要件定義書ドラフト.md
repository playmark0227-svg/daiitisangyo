## 1. プロジェクト概要

### 1.1 背景

- 北海道の一次産業（農業・漁業）の生産者は、高品質な生産物を持ちながらも、販路開拓・受発注業務がFAX・電話等のアナログ手段に依存しており、出品・受注のハードルが高い。
- 生産者には高齢者・デジタル操作に不慣れな層が多く、既存のECモール・受発注システムは操作が複雑で定着しにくい。
- 「その日獲れたもの・収穫したものを、その場で写真を撮ってすぐ売る」という一次産業特有の商流に対応したプラットフォームが存在しない。
- **市場に出回らない傷ついた魚（未利用魚）**等、規格外・従来販路に乗りにくい生産物の**新たな販路化**も重要な狙いとする。

### 1.2 プロジェクトの目的

1. 農業・漁業の生産者（出品者）が、**写真撮影と金額入力だけで即時に出品できる**、極限までシンプルな受発注アプリを提供する。
2. 発注者が毎日の入荷情報をプッシュ通知（LINE公式アカウントのような「今日の商品」お知らせ体験）で受け取り、締切時間内に簡単に発注・クレジットカード決済まで完結できる仕組みを構築する。
3. 管理者が出品・売上・手数料（マージン）を一元管理できる運営基盤を整備する。
4. まずはQRコード経由のクローズドな限定提供でテスト運用を行い、品質検証後に一般公開へ段階的に拡大する。

### 1.3 事業ビジョン（中長期構想）

- フェーズ1では農業・漁業の生産物を対象とするが、将来的には**道の駅・温泉施設の商品**等へ取扱いを拡大する。
- 北海道内の**市町村を巻き込んだ地域流通プラットフォーム**へと発展させ、地域経済圏全体をつなぐ基盤として、**数百億円規模の流通総額（GMV）を目指す**構想を持つ。
- 本フェーズのシステム設計は、この将来拡張（出品主体の多様化・取扱高の拡大・自治体連携）を阻害しないアーキテクチャとすることを方針とする（詳細は第6章「拡張性」参照）。

## 2. 用語定義

| 用語 | 定義 |
|---|---|
| 出品者 | 農業・漁業の生産者等、本プラットフォームに商品を出品するユーザー。 |
| 発注者 | 出品された商品を発注（購入）するユーザー。 |
| 管理者 | プラットフォーム運営会社。ユーザー・商品・売上・手数料を一元管理する。 |
| 定期商品 | 継続的・計画的に出品される商品（例：定番の農産物）。 |
| リアルタイム出品 | その場で写真を撮影し金額を入力するだけで即時出品する機能・運用形態（例：当日水揚げされた鮮魚）。 |
| マージン（手数料） | 管理者が設定するプラットフォーム手数料。**原価に手数料を加算して販売価格を自動算出する（加算モデル）。** |
| 締切時間 | 「何時までの発注で当日発送」等、当日出荷可否を分ける発注締切時刻。 |
| バッジ | 商品一覧上に表示するラベル表示。**「NEW」「人気」を基本とし、「朝どれ」等の追加ラベルはバッジマスタで拡張可能とする。** |
| 温度帯 | 商品の配送温度区分（冷凍／冷蔵／常温）。出品時に選択し、商品ページ・注文情報・発送指示に明示する。 |
| クローズドβ | QRコードを配布された限定ユーザーのみが利用できるテスト提供期間。 |
| 一般公開（GA） | クローズドβでの検証完了後、広く一般に開放するリリース段階。 |
| フェーズ1 | 今回開発（3〜4ヶ月）の初回リリース範囲。 |
| リリース1.x | フェーズ1（初回リリース）完了後の**早期機能拡充**。事業拡大を指すフェーズ2とは区別する。 |
| フェーズ2 | **事業拡大**（道の駅・温泉施設の取扱い、市町村・自治体を巻き込んだ横展開等）を指す段階。 |
| MoSCoW | 要件の優先度分類。Must（必須）/ Should（重要・推奨）/ Could（あれば望ましい）/ Won't（今回対象外）。 |
| GMV | 流通取引総額（Gross Merchandise Value）。 |

## 3. スコープ定義

本書では、**「フェーズ1」＝今回開発（3〜4ヶ月）**、**「リリース1.x」＝フェーズ1完了後の早期機能拡充**、**「フェーズ2」＝事業拡大（道の駅・温泉施設の取扱い、市町村・自治体を巻き込んだ横展開 等）** を指す。本章の定義を全ドキュメントのフェーズ用語の正とする。

### 3.1 フェーズ1（今回開発：3〜4ヶ月）

| 区分 | 内容 |
|---|---|
| 対象商材 | 北海道の一次産業（農業・漁業）の生産物 |
| 対象ユーザー | 管理者・出品者・発注者の3ロール |
| 提供形態 | 初期はQRコード経由のクローズド提供（限定ユーザーのみ）。テスト・検証後に一般開放。 |
| 出品機能 | 定期商品出品（テンプレート再出品を含む）、リアルタイム出品（写真＋金額のみ）、温度帯選択（冷凍/冷蔵/常温）、カテゴリ編集、バッジ、公開/非公開、リアルタイム在庫管理 |
| 価格・手数料 | 自動マージン計算（**加算モデル**：原価＋管理者設定手数料 → 販売価格自動算出、発注者へ総額表示） |
| 発注・決済 | 会員登録・認証、退会・アカウント削除、規約・ポリシー同意取得、通知許諾の事前説明、配送先住所管理、カート、簡単発注、送料計算（仕様は要確認）、締切時間表示・管理、Stripeクレジットカード決済、基本返金・注文取消処理 |
| 管理機能 | 3ロール権限管理、売上・商品一元管理ダッシュボード、取引明細CSV出力 |
| 通知 | プッシュ通知による「今日の商品」お知らせ |
| AI機能 | AI商品タイトル・説明文自動生成（**Should。手入力＋定型テンプレートへ縮退できる設計を必須とし、工数遅延時はリリース1.xへ繰延可能なフォールバック前提のフェーズ1目標**） |

### 3.2 フェーズ1完了後の拡充（リリース1.x）・フェーズ2以降（将来拡張）

> リリース1.x（フェーズ1完了後の早期機能拡充）とフェーズ2（事業拡大）は区別する。

| 区分 | 内容 |
|---|---|
| リリース1.x候補（フェーズ1完了後の早期拡充） | キーワード検索（F-011関連）、通知一覧タブ、出品者売上確認画面（F-209）、発注者起点のキャンセル・部分返金（F-111）、LINE公式アカウント連携、送り状発行システムとのAPI連携、インボイス本格対応（適格請求書の自動発行）、管理ダッシュボードの分析・グラフ拡充（F-202） |
| 商材拡大（フェーズ2） | 道の駅・温泉施設の商品の取扱い |
| 展開拡大（フェーズ2） | 一般公開後の機能拡張、市町村・自治体を巻き込んだ横展開 |
| 事業拡大（フェーズ2） | 数百億円規模のGMVを見据えたスケール対応（多地域展開、出品主体の多様化 等） |
| その他候補 | 決済手段の追加（請求書払い等）、多言語対応、物流連携の高度化 など（第10章の確認結果により整理） |

### 3.3 スコープ外（フェーズ1では実施しない事項）

- 道の駅・温泉施設商品の出品対応（フェーズ2以降）
- Stripeクレジットカード決済以外の決済手段（要否は第10章で確認）
- 物流・配送業務そのものの提供（配送手配の実務は出品者またはクライアント運用側の責任範囲とする想定（仮）。送料のシステム上の扱いはF-110で要件化し、仕様は第10章#1で確認）
- **酒類の取扱い（仮）**：酒類のEC販売には通信販売酒類小売業免許が必要となるため、フェーズ1では酒類を取扱対象外とする（仮）。フェーズ2（道の駅商品等）で取扱う場合は免許要否を含め別途確認する（第10章#6参照）。

## 5. 機能要件一覧

優先度は MoSCoW（Must=必須 / Should=重要 / Could=可能なら / Won't=今回対象外）で付与。

### 5.1 出品・商品管理

| 要件ID | 機能名 | 要件概要 | 優先度 |
|---|---|---|---|
| F-001 | 定期商品出品 | 出品者が定番・継続商品を商品名、写真、価格（原価）、在庫数、温度帯（冷凍/冷蔵/常温）、カテゴリ等を設定して出品できる。**定期商品テンプレートからの再出品（数量のみ入力）を含む。** | Must |
| F-002 | リアルタイム出品 | **写真撮影＋金額（原価）入力のみ**で即時出品が完了する簡易出品フロー。当日入荷品等をその場で出品できる。入力項目を極小化する。**温度帯（冷凍/冷蔵/常温）の選択を必須とするが、3択の大ボタン、テンプレート・前回値からのデフォルト適用により追加タップは原則ゼロとする。** | Must |
| F-003 | カテゴリ編集 | 商品カテゴリの作成・編集・並び替えができる（管理主体は管理者を想定（仮）、出品者権限の付与範囲は要確認）。 | Must |
| F-004 | バッジ表示（NEW/人気） | 商品に「NEW」「人気」等のバッジを表示できる。**「NEW」「人気」を基本とし、「朝どれ」等の追加ラベルはバッジマスタで拡張可能とする。** 付与ルール（手動/自動：新着日数・売上数連動等）および出品者への付与権限は要確認（第10章#15）。 | Should |
| F-005 | 公開/非公開切替 | 商品ごとに公開/非公開を切り替えられる。非公開商品は発注者に表示されない。 | Must |
| F-006 | リアルタイム在庫管理 | 商品ごとに在庫数量を設定し、**指定数量以上は売れない（売り越しを防止する）制御**を行う。発注確定と同時に在庫を即時減算し、在庫0で自動的に売り切れ表示とする。同時アクセス時の二重販売を防止する排他制御を含む。 | Must |
| F-007 | 自動マージン計算（加算モデル） | 出品者が入力した**原価に、管理者が設定した手数料（マージン率/額）を加算し、販売価格を自動算出**する。出品者は原価入力のみで価格設定が完結し、**発注者には総額（税込販売価格）を表示**する。**Stripe Connect採用時の対応関係**：発注者は販売価格を支払い、マージン相当額を `application_fee_amount` としてプラットフォームが受領、原価相当額が出品者に帰属する。なお原価の税込/税抜の扱い・端数処理は確認事項（第10章#13・#14、税理士確認）とする。 | Must |
| F-008 | AI商品タイトル自動生成 | 商品写真・簡易入力情報から商品タイトルをAIで自動生成する。**手入力＋定型テンプレートへ縮退できる設計を必須とする。** 工数遅延時はリリース1.xへ繰り延べ可（フォールバック前提のフェーズ1目標）。 | Should |
| F-009 | AI商品説明文自動生成 | 商品写真・簡易入力情報から商品説明文をAIで自動生成する。生成文は出品者が確認・修正のうえ確定する運用を推奨。**手入力＋定型テンプレートへ縮退できる設計を必須とする。** 工数遅延時はリリース1.xへ繰り延べ可（フォールバック前提のフェーズ1目標）。 | Should |
| F-010 | 商品写真管理 | スマートフォンのカメラ起動・撮影・アップロードを出品フロー内で完結できる。 | Must |
| F-011 | 商品一覧・詳細表示 | 発注者向けに商品一覧（カテゴリ別・バッジ表示含む）と商品詳細（価格・在庫・締切時間・温度帯）を表示する。 | Must |

### 5.2 発注・決済・アカウント

| 要件ID | 機能名 | 要件概要 | 優先度 |
|---|---|---|---|
| F-101 | カート | 複数商品をカートに入れ、まとめて発注できる。カート・注文確認画面では商品代金に加え送料（F-110）を計上・表示する。 | Must |

## 9. 前提条件・制約事項

### 9.1 前提条件

1. 開発期間は3〜4ヶ月とする（フェーズ1）。
2. 決済はStripeを利用し、クライアント名義のStripeアカウントを開設できること（審査通過を含む）。
3. 初期リリースはQRコード経由のクローズド提供とし、一般公開はテスト後にクライアント判断で行う。
4. 商品マスタ初期データ・出品者の初期リスト・QR配布先はクライアントが準備する（仮・要確認）。
5. 利用規約・特定商取引法表記・プライバシーポリシー等の法務文書はクライアント側で内容確定する（作成支援の範囲は要確認）。
6. 配送業務（梱包・出荷・配送手配）の実務は出品者またはクライアント運用側が担い、システムは注文・配送先情報の提供と送料計上（F-109/F-110）までを担当範囲とする（仮・第10章#1で確認）。

### 9.2 制約事項

1. 3〜4ヶ月の開発期間内に収めるため、機能はMoSCoWのMustを最優先で実装し、Should（AI生成機能等）は工数状況によりリリース1.xへ繰り延べる場合がある。**なお、基本返金・注文取消処理（F-107）はカード決済を伴うECのローンチ必須機能としてMustに位置付け、繰り延べ対象としない。**
2. 決済手数料はStripe所定の料率に従う。**出品者への入金はStripeが提供する入金スケジュール設定に従う（実装時点の最新仕様を確認）。**
3. LINE連携を採用する場合、LINE公式アカウント・Messaging APIの利用料金体系（配信数課金）と審査に依存する。
4. AI生成機能の出力品質はAIモデルに依存するため、出品者による確認・修正を前提とした運用とする。
5. 生鮮品の品質・配送・在庫の現物差異など、システム外のオペレーションに起因する事象はシステム保証の対象外とする（運用ルールで補完。第10章参照）。
6. 代金収受・分配のスキーム（第10章#5）の法務確認結果によっては、決済フロー・入金フローの設計に制約が生じる可能性がある。
